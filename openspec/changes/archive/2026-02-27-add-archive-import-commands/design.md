## Context

`fs` 工具当前支持通过 YAML 文件管理多个主机配置，配置文件存储在 `~/.fs/hosts/` 目录下。每个主机配置对应一个独立的 YAML 文件。用户在环境迁移、团队协作或备份恢复时，需要手动处理多个配置文件，缺乏统一的配置管理机制。

## Goals / Non-Goals

**Goals:**
- 提供配置的打包和导出功能，支持单文件分发
- 实现配置的导入功能，支持批量配置恢复
- 确保配置导入时的数据完整性和安全性
- 提供友好的冲突处理机制

**Non-Goals:**
- 不处理配置文件的加密存储
- 不提供配置版本管理功能
- 不支持部分配置的增量导入
- 不实现配置的远程同步功能

## Decisions

### 打包格式选择
**决策**: 使用 tar.gz 格式作为打包文件格式
**理由**: 
- tar.gz 是标准的压缩格式，跨平台兼容性好
- Go 标准库原生支持，无需引入额外依赖
- 支持保留文件元数据和目录结构
- 文件大小相对较小，便于传输

### 配置冲突处理策略
**决策**: 采用交互式选择 + 命令行参数控制的混合策略
**理由**:
- 默认交互式询问用户处理方式（覆盖、跳过、重命名）
- 提供 `--force` 参数强制覆盖，`--skip-existing` 参数跳过已存在配置
- 满足不同使用场景的需求

### 文件结构设计
**决策**: 打包文件内包含配置文件和元数据信息
**结构**:
```
fs-config-archive/
├── config/
│   ├── host1.yaml
│   ├── host2.yaml
│   └── ...
├── metadata.json
└── VERSION
```
**理由**: 
- 分离配置文件和元数据便于管理和验证
- metadata.json 包含打包时间、版本等信息
- VERSION 文件确保兼容性检查

### 安全性考虑
**决策**: 实现基本的文件验证机制
**措施**:
- 验证打包文件的完整性
- 检查配置文件格式的有效性
- 导入前预览配置内容
- 限制可导入的文件类型和大小

## Risks / Trade-offs

**配置覆盖风险** → 提供多种冲突处理选项和明确的用户提示
**打包文件损坏** → 实现校验和验证机制
**大文件处理性能** → 限制单个打包文件大小，提供进度显示
**跨版本兼容性** → 在 metadata 中包含版本信息，实现版本检查

## Migration Plan

6. 测试各种使用场景
